LESSON 5: Custom Commands and Automation
========================================

You hate writing PR descriptions. You've said it, you know it, and every time you finish a feature you have to sit down and write the thing anyway. That changes today.

Lesson 5 is about custom commands — and your PR description problem is the perfect candidate.

---

THE TASK (Intermediate level)
------------------------------

Here's what I want you to do:

Create a custom command called `pr-description` that, when you invoke `/pr-description`, automatically generates a solid PR description based on your current branch's diff.

Here's how to set it up:

1. In your project, create the directory `.claude/commands/` if it doesn't exist.

2. Create a file: `.claude/commands/pr-description.md`

3. Write the prompt inside it. Here's a starting point you can adapt:

---

```
Look at the git diff between the current branch and main (run: git diff main...HEAD).

Based on the diff, write a pull request description with:
- A one-line summary of what changed and why
- A "Changes" section listing the key modifications (3-5 bullets, technical but readable)
- A "Testing" section describing what to check when reviewing
- Any migration steps or breaking changes if applicable

Write it as if you're a senior engineer explaining the change to a teammate. Skip filler phrases. Be specific.
```

---

4. Now invoke it in Claude Code: `/pr-description`

Watch what happens.

---

A FEW THINGS TO TRY AFTER IT WORKS
------------------------------------

- The command is general enough to work on any branch. Try it on a branch with a small change and one with a bigger refactor — does the output scale well?

- If the output is too long or too short, edit the prompt. The file is yours. That's the whole point.

- Want to customize it per-project? Keep the file in `.claude/commands/` inside that repo. Want it everywhere? Move it to `~/.claude/commands/` in your home directory and it'll show up in any project.

---

WHAT YOU'RE ACTUALLY BUILDING
-------------------------------

This command turns a 15-minute task you dread into a 10-second invocation. But the bigger shift is this: you're now treating your Claude setup as code. The command file is a productivity artifact that lives in your repo, gets committed, and can be improved over time — just like any other piece of infrastructure.

Every developer on your team could share this command. Or you could keep it to yourself. Either way, it compounds.

---

DEBRIEF
--------

You just automated something you hated doing. That's the shift lesson 5 is about — going from user to builder. Custom commands are the first place where Claude Code becomes yours, not just the default tool.

The concept underneath this: commands live in `.claude/commands/` as markdown files. Filename = command name. Content = the prompt. Claude can use any tools it has access to (including Bash, file reading, git) when running the command. You're not writing a shell script; you're writing an instruction that can trigger any capability Claude has.

QUICK REFERENCE
---------------

Q: Where do command files live?
A: `.claude/commands/` in your project (project-specific) or `~/.claude/commands/` (available everywhere).

Q: What format?
A: Markdown. Filename becomes the command name. Content is the prompt.

Q: Can commands use $ARGUMENTS?
A: Yes. Use `$ARGUMENTS` in your prompt to accept input when invoking. Example: `/pr-description fix/auth-bug` — and your prompt references `$ARGUMENTS` where the branch or ticket name should go.

Q: What's the difference between this and CLAUDE.md?
A: CLAUDE.md is always-on context — it shapes every conversation. Commands are on-demand — you invoke them intentionally. Permanent preferences go in CLAUDE.md; repeatable tasks go in commands.
